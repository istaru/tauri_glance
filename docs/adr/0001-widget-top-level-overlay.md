# Windows widget 用顶层叠加窗口而非 SetParent 嵌入任务栏

Windows 没有受支持的"任务栏嵌入"API。曾用 `SetParent` 把 widget 变成 `Shell_TrayWnd` 的 `WS_CHILD` 子窗口做到"真嵌入"，但**跨进程子窗口收不到鼠标/pointer 事件 → 无法拖动**。故改为独立的 `always_on_top` 顶层窗口，用屏幕坐标叠在任务栏那一条上：牺牲"真嵌入"的原生观感，换回鼠标事件与拖动定位能力。

## 权衡的选项

- **SetParent 子窗口**：视觉上真嵌进任务栏，但跨进程子窗口拿不到鼠标事件，用户无法拖动、无法交互。
- **顶层叠加窗口（选定）**：不是真嵌入，只是置顶叠在任务栏那一行上；能正常收 pointer 事件，可拖动。

拖动被定为**正式功能**（见 [CONTEXT.md](../../CONTEXT.md) 的「悬浮窗」「锚点」）：Win11 无嵌入 API，叠加窗口的位置可能和用户的托盘图标/时钟相撞，所以让用户能微调位置是刚需，不是可有可无的副产品。

## 后续影响

- **定位只在启动时算一次**：托盘图标增减、时钟格式变化、换分辨率导致 `^` 箭头移位后，widget 不会自动重贴，需重启或手动拖动。这是刻意选的简单方案（相对"每 tick 跟箭头走 + 存偏移量"）。
- **拖动持久化存绝对屏幕 x**（`widget_x` 文件），启动时 clamp 到任务栏 x 范围内。换分辨率/多屏后旧值可能相对箭头漂移。
- **拖动锁行**：只允许水平移动（保持在任务栏那一行），y 恒定，前端只存 x。
